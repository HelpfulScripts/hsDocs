  69## Creating the Documentation DocSet

  70Follow the instruction for `typedoc` in commenting the code and set the `json` option to create a 

  71json file containing the documentation. See the Configuration section below for details.

  72

  73In addition to the documented source files, `hsDocs` recognizes a special **`overview.ts`** file 

  74that will be displayed as a project overview. 

  75For this to work, `typedoc` requires the file to have two separate comment entries.

  76If the second comment is missing, `typedoc` will not generate a comment in the DocSet.

  77

  78DocSets are shown as menu tabs across the top of the window. To include a DocSet in the menu, ensure that the 

  79corresponding documentation `.json` file is copied into the `data` folder inside the folder that holds the serving html file.

  80Then edit the `index.json` file to include the documentation file name in the "docs" section. If no `index.json` file is found, 

  81`hsDocs` will attempt to load a directory listing of the `data` file and include all encountered `*.json` files as DocSet. 

  82This requires enabling the http server to provide a directory listing for the `data` folder.

  83

  84### Modules

  85Each source file is considered to be a `module`. Modules are shown in the left-hand overview panel of the main `hsDocs` window.

  86Alternatively, a `@module` statement in the summary comment at the top of a file will create, or add to, a module of the specified name.

  87This encourages to separate code into files without cluttering the overview.

  88

  89### Linking across the docsets

  90Links to items across the docsets can be placed via a link directive: 

  91- `{@link [docset:]module.item` linked text`}`, 

  92

  93where

  94 - *docset* optional; the name of the docset. If omitted, a link within the current docSet will be created

  95 - *module* is name of the module to link to; 

  96   if *module* is `overview`, or unspecified, the link will point to the docset overview page.

  97 - *item* is the name, or period-separated path, of the item within the module. 

  98 - *linked text* is displayed with a link to path within module

  99

 100Examples:

 101- `{@link overview` this hsDocs overview`}` -> {@link overview this hsDocs overview}

 102- `{@link hsDocs:` this hsDocs overview`}` -> {@link hsDocs: this hsDocs overview}

 103- `{@link hsDocs:DocSets.DocSets.add` the `add` function`}` -> {@link hsDocs:DocSets.DocSets.add the `add` function}

 104- `{@link DocSets.DocSets.add` the `add` function`}` -> {@link DocSets.DocSets.add the `add` function}

 137## Configuring typedoc

 138As an example, we use grunt-typedoc with the following configuration:

 139

 140### tsconfig.json:

 141A `tsconfig.json` file seems to be required by typedoc. The file can be empty

 142

 143### Gruntfile:

 144module.exports = function(grunt) {

 145    ...

 146    typedoc: {

 147        code: {

 148            options: {

 149                tsconfig: 'typedoc.json',

 150                json:   './docs/hsDocs.json',

 151                out:    './docs',

 152                mode:   'modules',

 153                name:   'hsDocs'

 154            }

 155            src: ['src/*.ts', '!src/*.spec.ts']

 156        }

 157    },

 158

 159   ...

 160   grunt.loadNpmTasks('grunt-typedoc');

 161   ...
